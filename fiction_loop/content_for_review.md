## 5. CANONICAL PROBLEM STRUCTURES — what each gate IS as a problem

One template per operation: the abstract shape of a gate that teaches it
(`canonical_problem_structure`: unknown / data / condition) plus the
`canonical_correct_approach`. The Writer dresses these in concrete objects, city,
and character; the shape itself never varies. **This is the section that most
deserves your careful read.**

1. **Identify unknown / data / condition** — *unknown:* never stated; the room
   presents objects and a mechanism and lets the solver assume what is wanted.
   *data:* fully visible, slightly too orderly — invites action before understanding.
   *condition:* explicit but easy to skip past. *approach:* stop at the entrance;
   name what is wanted, what is given, and what binds them — before touching anything.
2. **What is missing** — *unknown:* an absence; the room is complete except for one
   thing that should be there and is not. *data:* an almost-complete set whose
   completeness invites cataloguing. *condition:* closure responds to the absence
   being named, not to the present things being arranged. *approach:* ask what should
   be here and is not; search the gaps, not the contents.
3. **Separate the parts of the condition** — *unknown:* clear. *data:* clear.
   *condition:* compound — several clauses fused into what reads as one requirement;
   taken whole, it stalls. *approach:* split the condition into clauses; test each
   alone; find which one actually binds.
4. **Did you use all the data?** — *unknown:* clear. *data:* N elements, one of which
   the obvious path never needs — and the obvious path fails for exactly that reason.
   *condition:* satisfiable only when every given element plays a role. *approach:*
   when the partial path stalls, inventory given-versus-used; bring in the untouched
   element.
5. **Can you check the result?** — *unknown:* clear, with a plausible solution easy to
   produce. *data:* supports two candidates — one correct, one method-correct but
   wrong. *condition:* contains a built-in way to test a candidate against the
   original requirement. *approach:* do not stop at "the method worked"; test the
   result against the condition itself.
6. **Can you derive the result differently?** — *unknown:* already found once, by a
   long route. *data:* contains a shorter route, visible only in retrospect.
   *condition:* demands the result be reached or confirmed by a second, independent
   route. *approach:* after solving, ask whether a cleaner derivation exists; find it.
7. **Look at the unknown** — *unknown:* of a recognizable *category* (a quantity, a
   place, a person) — naming the category activates memory. *data:* abundant and
   seductive; starting from it leads into a maze. *condition:* links data to unknown
   only when read from the unknown's side. *approach:* stand with the unknown; ask
   what kind of thing is wanted and what has ever produced that kind of thing.
8. **Do you know a related problem?** — *unknown:* the same *kind* as one the solver
   has met before. *data:* dressed in a surface unlike any previous gate.
   *condition:* structurally identical to the ancestor problem's. *approach:* search
   memory for a problem with the same or similar unknown; the surface differs, the
   skeleton doesn't.
9. **Here is a problem related to yours and solved before** — *unknown:* reachable by
   adapting a known solution, not by re-deriving. *data:* includes a bridge element
   that maps the old solution onto the new problem. *condition:* satisfied by the old
   method or result — but only after deliberate adaptation. *approach:* hold the
   recalled problem next to this one: use its result? its method? what must be added
   to make it usable?
10. **Analogy — simpler analogous problem** — *unknown:* too complex to attack
    directly. *data/condition:* relations that survive simplification — a smaller
    problem with identical structure can be built. *approach:* construct the simplest
    problem with the same relations; solve it; carry the method back.
11. **Looking back / transfer** — *unknown:* not the gate's — the solver's own method
    is the unknown. *data:* the just-finished solution, still present and readable.
    *condition:* nothing forces staying; leaving costs nothing today and everything
    at the next gate. *approach:* stay; ask what was actually done; name where else
    it applies.
12. **Decompose condition / keep part drop part** — *unknown:* unreachable under the
    full condition. *data:* sufficient for a partial problem. *condition:* compound;
    the whole resists, but one part alone determines something solid. *approach:*
    keep one clause, drop the rest; solve what the partial condition determines;
    reintroduce the clauses one by one.
13. **Introduce auxiliary elements** — *unknown:* connected to the data only through
    something *not given*. *data:* complete but disconnected — two shores, no bridge.
    *condition:* satisfiable only after the solver adds an element of their own.
    *approach:* add what was not given — the line, the mark, the sub-unknown that
    makes a known method applicable.
14. **Can you use the result?** — *unknown:* belongs to the *next* problem; this
    gate's result is a tool. *data:* a second chamber restates the first chamber's
    result in new clothes. *condition:* the second chamber is unsolvable from scratch
    in the time available — and instant with the first chamber's result. *approach:*
    treat every result as an asset; ask what it now unlocks.
15. **Auxiliary problem (invented)** — *unknown:* no foothold; the forward path
    exhausted. *data:* useless to the original problem directly, but sufficient for a
    different problem the solver could pose. *condition:* the original yields only
    via the invented side-problem — and the detour visibly costs time and may fail.
    *approach:* deliberately pose a different problem as a stepping stone, knowing
    the risk; work it; return with what it yields.
16. **Generalisation** — *unknown:* a specific instance whose specificity hides the
    structure (source: "passing from the consideration of one object to a set
    containing that object"). *data:* peculiarities that look load-bearing and
    aren't. *condition:* an instance of a general class with a reachable class
    solution. *approach:* ask what this is a case *of*; solve the class; step back
    down to the instance.
17. **Specialisation** — *unknown:* general and sprawling; no purchase anywhere
    (source: "passing from a given set to a smaller set, or of just one object").
    *data:* contains one simplest or extreme case that can actually be worked.
    *condition:* whatever holds must hold in the simplest case — which shows the
    pattern or kills the approach. *approach:* contract to the simplest case; solve
    it; use it as foothold or counterexample.
18. **Working backwards / analysis** — *unknown:* unreachable from the data side;
    every forward move dead-ends. *data:* many forward paths, all divergent — from
    the goal side, one path is visible. *condition:* a chain of dependencies legible
    only in reverse. *approach:* assume it solved; stand at the end; ask what must
    have been true immediately before, and step back until known ground.
19. **Variation / restatement** — *unknown:* as stated, the wrong unknown — the
    stated problem is solvable and solving it changes nothing. *data:* contains a
    remainder the stated problem never touches. *condition:* the true condition
    differs from the apparent one; closure responds only to the restated problem.
    *approach:* ask whether the stated problem is the real problem; restate until the
    condition matches what the gate actually requires.
20. **Inventor's Paradox** — *unknown:* specific, and harder *because* specific
    (source: "the more ambitious plan may have more chances of success… the more
    general problem may be easier to solve"). *data:* insufficient for the narrow
    question, exactly right for the broader one. *condition:* the general problem's
    solution covers the specific case; the specific case alone offers no structure.
    *approach:* enlarge the problem on purpose; answer the bigger question; the
    specific answer falls out.
21. **Reductio ad absurdum** — *unknown:* a certainty — something to prove impossible
    or necessary; no direct proof available. *data:* consistent with the false
    assumption at first; the contradiction sits several honest steps deep.
    *condition:* assuming the opposite and following it faithfully must break against
    a given fact. *approach:* assume what you believe false; follow it precisely;
    the break is the proof.
22. **Heuristic reasoning vs proof** — *unknown:* reachable by a plausible path that
    cannot yet be verified (source: heuristic reasoning is "provisional and plausible
    only, whose purpose is to discover the solution"). *data:* enough for a good
    guess now; enough for verification only separately, later. *condition:* acting on
    the guess is necessary (time), and verifying it is also necessary (what follows
    punishes unverified confidence). *approach:* commit to the provisional path while
    separately tracking what remains unverified; do both, confuse neither.
23. **Subconscious work** — *unknown:* will not yield to continued effort; every
    conscious approach exhausted (source: "there is a limit beyond which we should
    not force the conscious reflection… take counsel of your pillow"). *data:* fully
    absorbed already; nothing new to gather. *condition:* the connection forms only
    after effort stops. *approach:* stop entirely; leave; return — the right question
    is present on return.
24. **Auxiliary problem chains / Pappus** — *unknown:* separated from the data by
    several equivalent reformulations; no single reduction suffices. *data:* supports
    the *last* problem in a chain, not the first. *condition:* each reduction must
    preserve equivalence, and closure requires walking the chain back (synthesis)
    after building it forward (analysis). *approach:* reduce problem to problem until
    one is solvable; then retrace the chain backward to deliver the original.
